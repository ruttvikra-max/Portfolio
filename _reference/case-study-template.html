The Invisible Problem

Design drift is not a bug.
It's a lifecycle.

A slow, compounding breakdown that gets worse every time a team ships without looking back.

The compounding cycle — it never naturally stops

Design System Created → Features Ship → Local Overrides → Detached Components → Design–Dev Mismatch → Manual Fixes → System Authority Weakens ↺
01
Detached Components

Components break away from the source library. Changes stop propagating — every edit becomes a manual patch.

02
Token Overrides

Design tokens get hardcoded. Visual consistency breaks silently — one screen at a time.

03
Naming Divergence

Inconsistent naming patterns break handoff, fail automation, and make the system a maze.

04
Design–Dev Misalignment

What ships is no longer what was designed. The gap grows invisibly. Debt compounds quietly.

The Key Decision

Naming the problem wasn't enough. It had to be measurable.

Every designer I spoke to could name the symptoms. But naming a problem and defining it precisely enough that a system can score it are very different things. That gap is where this project was born.

"The hardest part wasn't identifying what was wrong — it was defining the problem precisely enough that a system could score it. The shift from observation to measurable definition is where the four pillars were born."

Easy Answer
"Detect when components are wrong"

Vague, unmeasurable. An AI can't score what it can't define. Would have produced a tool with no real output.

Right Answer
Define specific, scoreable dimensions

Each pillar became a precise detection target — not a category, but a measurable compliance state. Specificity makes scoring possible.

Hardest Call
v1 is analysis only — no auto-fix

Professionals confirmed it: trust in the analysis must come before automation can be introduced.

The Solution

Four pillars. One score.
Built inside Figma.

A Figma Plugin that detects, measures, and reports on design system health through four precisely defined pillars of compliance.

01
Component Integrity

Detects detached instances, unlinked components, and structural overrides across the entire Figma file. Ranked #1 priority by professionals during usability testing — the most urgent, highest-frequency pain point in practice.

Detached InstancesUnlinked ComponentsStructural Overrides
02
Token Compliance

Flags misused, missing, or hardcoded design tokens across color, spacing, and typography. Surfaces every place where raw values have replaced semantic tokens.

Color TokensSpacing ViolationsTypography TokensHardcoded Styles
03
Naming Governance

Detects violations of architecture-based naming conventions. Critical for handoff quality, automation reliability, and team consistency — naming divergence is invisible until it breaks the pipeline.

Naming ViolationsHandoff QualityAutomation Reliability
04
System Drift Score

A single quantified health score aggregating all three pillars. Outputs a compliance percentage, severity ranking, and file health overview — the number that makes invisible drift visible to engineers, PMs, and stakeholders.

Compliance %Severity RankingFile Health Overview
The Pipeline

Four steps. Figma as the data source.

The plugin reads the active file directly through the Figma Plugin API — no imports, no external uploads. Analysis happens inside Figma, on the file being designed.

STEP 01
Ingest

Connect to the Figma file and linked design system. Establish the source of truth — what the system is supposed to look like.

STEP 02
Scan

Traverse the full component tree, token usage, and naming patterns across every layer. Nothing sampled — everything checked.

STEP 03
Detect

Flag issues across all four pillars. Rank by severity and estimated impact. Surface what needs attention first.

STEP 04
Report

Output a full drift report with ranked issues, compliance scores, and a system health overview — analysis only, no automated changes.

Usability Testing

Testing didn't validate the design. It changed it.

Prototype sessions with working professionals revealed something important: feature prioritisation that made sense on paper didn't match what caused the most pain in practice.

Finding 01
Component Integrity moved to #1

Professionals flagged detached components as their most urgent pain — above token violations and naming issues. The pillar hierarchy shifted to reflect this.

Finding 02
Report structure re-sequenced

What users see first changed — from what seemed logical on paper to what actually causes the most pain. Context, not assumption, drove the final sequence.

Finding 03
"Analysis only" approach confirmed

Every professional validated the v1 decision to report rather than auto-fix. Trust in the analysis must come before automation.

What I Learned

What changed me,
not just the work.

Three months of working on a problem that's invisible until it becomes expensive. What I'm taking forward isn't just a portfolio piece — it's a shift in how I think about the relationship between design systems and the teams that depend on them.

Learning 01
Definition is design work

The hardest decision wasn't the interface — it was defining "drift" precisely enough that a system could measure it. Measurability is not a technical problem. It's a design problem.

Learning 02
Constraints are part of the solution

"Analysis only" in v1 felt like a limitation. It turned out to be strategic — and users confirmed it. Knowing when not to build something is as important as knowing what to build.

Learning 03
Where the product lives changes everything

Building the standalone app first wasn't a mistake — it was necessary to see the problem clearly. The pivot to a plugin was the product finding its right home.
